1. Introduzione

1.1 Cos'è exa-dune?

exa-dune è un framework di penetration testing scritto in Bash (~13.500 righe), progettato per ambienti IoT e VoIP. Integra e coordina oltre 40 tool di sicurezza di Kali Linux attraverso un'interfaccia unificata a subcomandi, automatizza il ciclo completo di assessment e genera report in italiano con scoring CVSS 3.1.

Avvertenza legale
Questo strumento è destinato esclusivamente a sistemi di tua proprietà o per i quali hai esplicita autorizzazione scritta. L'uso non autorizzato costituisce reato penale in Italia e in tutti i paesi dell'UE.

1.2 Cosa può fare exa-dune v2.8.0?

  • 35+ subcomandi — dalla scansione alla post-exploitation, copertura completa
  • Assessment automatico a 4 fasi — un solo comando auto esegue scan, enum, attacchi e report PDF
  • 32 tecniche di bypass firewall — ACK, FIN, frammenti IP, source port spoofing, UPnP, SIP-ALG, port knocking, IoT vendor ports e altro
  • Enumerazione multi-protocollo — HTTP, HTTPS, SIP, SIP-WS, SSH, Telnet, FTP, RTSP, ONVIF, SNMP, TFTP, MQTT, SMB, UPnP, Modbus, BACnet, EtherNet/IP, CoAP, IPMI, DNS, RTMP, RDP, VNC, LDAP, Asterisk AMI, DLNA, STUN, MGCP, H.248, S7comm, DNP3
  • CVE database integrato — Hikvision, Dahua, Axis, Reolink, AudioCodes MP20x, Commend ET962
  • Credential store — credenziali trovate salvate e correlate cross-service
  • Sistema checkpoint/resume — riprendi assessment interrotti senza perdere il lavoro svolto
  • Pipeline automatizzata — scan → suggest → attacchi → post-exploit → PDF
  • Wordlist embedded — funziona anche senza SecLists installato

La logica di branching nella FASE 1 è a tre rami:

  • Ramo 1 (normale): scan trova porte/servizi → si procede direttamente con la FASE 2
  • Ramo 2 (bypass): scan non trova nulla → si tenta il bypass firewall con 32 tecniche → se una funziona si torna alla FASE 2
  • Ramo 3 (fallback): bypass fallisce completamente → si salta direttamente alla FASE 4 con un report "no access"

3.3 Credential Store

Ogni credenziale trovata durante il test viene salvata in .exa_creds.json per servizio, host e porta. Al termine della FASE 3, exa-dune esegue una correlazione cross-service: testa le credenziali trovate su tutti gli altri servizi aperti del target.

5. Comando AUTO — Assessment Completo a 4 Fasi

auto è il comando principale di exa-dune: esegue un assessment completo e automatizzato di un target, dall'identificazione iniziale fino alla generazione del report PDF, tutto in un singolo terminale con output in tempo reale.

5.0 Sintassi

exa-dune auto <TARGET> [OPTIONS]

# TARGET può essere un indirizzo IP o un hostname
exa-dune auto 192.168.1.10
exa-dune auto audiocodes.lab.local

5.0.1 Opzioni specifiche di auto

OpzioneDescrizione
--stealthModalità stealth: scan lento, rate ridotto, timing nmap T2
--timeout NTimeout in minuti per ogni modulo (default: 60). Ogni modulo viene wrappato con timeout N*60s exa-dune ...
--userlist FILEWordlist utenti custom per brute-force (propagata a tutti i moduli)
--passlist FILEWordlist password custom (propagata a tutti i moduli)
--output DIRDirectory di output. Se non specificata: /root/pentest/exa-dune-TARGET_TIMESTAMP/
--resume DIRRiprende assessment da una directory esistente, saltando i moduli già completati (usa .exa_checkpoint)
--dry-runMostra i comandi senza eseguirli, usando un piano attacchi demo

5.1 FASE 1 — SCAN

La FASE 1 è suddivisa in tre sotto-fasi: pre-scan (ricognizione passiva/veloce), scan principale (nmap completo), post-scan (MAC vendor lookup).

Pre-scan (9 operazioni)

#OperazioneToolScopo

2Passive ARP sniff (10s)tcpdumpCattura trame ARP senza inviare pacchetti — ottiene MAC address in modo stealth
3ICMP ping checkpingDecide se usare -Pn negli scan successivi. Se ICMP bloccato: imposta NMAP_OPTS_EXTRA=-Pn
4TraceroutetracerouteConta gli hop, rileva NAT/firewall intermedi confrontando ultimo hop con target IP

6WS-Discovery probeexa-dune wsd (UDP 3702)Telecamere e device embedded annunciano servizi ONVIF/WSDL
7SNMP quick probesnmpgetProva community public/private/community/admin su UDP 161. Se risponde: salva in snmp_quick.txt

9SSL certificate grabopenssl s_clientCN/SAN/O del certificato :443 contengono spesso vendor e modello device

Scan principale

  • nmap fast — top 1000 porte TCP, -sV -sC -O, service detection + script default
  • nmap full — tutte le porte TCP -p- (saltabile con --no-full)
  • nmap UDP — top 20 porte UDP più comuni
  • ARP scan — rileva MAC address e vendor OUI
  • searchsploit — ricerca CVE dal XML nmap: searchsploit --nmap nmap_fast.xml

Post-scan

Dopo lo scan, exa-dune effettua il MAC vendor lookup dal file arp.txt (prodotto da arp-scan) e aggiunge il risultato al contesto di pre-scan per migliorare il riconoscimento del device nella FASE 2.

Logica a 3 rami

Al termine della FASE 1, exa-dune controlla se ha trovato punti deboli verificando: porte aperte TCP/UDP, MAC/vendor rilevato, hostname DNS, risposta HTTP HEAD, certificato SSL, mDNS/WSD findings, risposta SNMP, ARP passivo, traceroute arrivato al target, match searchsploit.

  • Se almeno uno di questi è positivo → FASE 2 ENUMERAZIONE
  • Se nessuno → tenta bypass firewall (32 tecniche)
  • Se bypass fallisce → FASE 4 REPORT con no_access_summary.txt

5.3 FASE 2 — ENUMERAZIONE

La FASE 2 lancia automaticamente i moduli di enumerazione appropriati in base alle porte aperte trovate nella FASE 1. Prima di tutto, exa-dune aggrega il contesto del pre-scan (HTTP Server header, SSL CN/O, SNMP sysDescr, DNS hostname, mDNS/WSD findings, MAC OUI, SNMP OID enterprise) per rilevare il vendor del device.

Rilevamento vendor automatico

Il contesto pre-scan viene confrontato con pattern per identificare: audiocodes, commend, gai-tronics, axis, reolink, hikvision, dahua. Anche l'OID SNMP enterprise viene mappato al vendor (es. 1.3.6.1.4.1.591.* = AudioCodes).

NSE scripts mirati post-discovery

Dopo i moduli standard, exa-dune esegue NSE scripts Nmap mirati sui servizi trovati:

  • HTTP: http-auth-finder, http-default-accounts, http-methods, http-headers, http-robots.txt
  • FTP: ftp-anon, ftp-bounce, ftp-syst
  • SSH: ssh2-enum-algos, ssh-hostkey, ssh-auth-methods
  • SMB: smb-vuln-ms17-010, smb-vuln-ms08-067, smb-security-mode, smb2-security-mode
  • SNMP: snmp-brute, snmp-info, snmp-interfaces
  • VNC: vnc-info, realvnc-auth-bypass
  • RTSP: rtsp-url-brute, rtsp-methods
  • SIP: sip-enum-users, sip-methods
  • RDP: rdp-enum-encryption, rdp-vuln-ms12-020
  • MQTT: mqtt-subscribe

Banner grab su porte sconosciute

Per ogni porta TCP aperta non coperta da moduli specifici, exa-dune esegue: nmap -sV sulla porta, banner grab raw con nc, HTTP HEAD (http e https), e infine searchsploit sul banner/versione rilevata per trovare CVE noti.

Fallback intelligente (nessuna porta trovata)

Se nessun modulo si è attivato (ran_enum=0), exa-dune usa i dati del pre-scan per decidere cosa fare: se HTTP HEAD ha risposto, lancia web; se SNMP ha risposto, lancia snmp; se vendor è camera/voip, lancia i moduli appropriati. Come ultima risorsa: web:80, sip, snmp, ssh.

Device profile consolidato

Al termine della FASE 2, viene generato device_profile.txt con: target, hostname DNS, vendor rilevato, tipo device (camera/VoIP), MAC e vendor OUI, SSL CN/O, porte aperte TCP/UDP, servizi nmap -sV, risposta SNMP, conteggio findings per severità (CRITICAL/HIGH/MEDIUM/LOW), top findings, credenziali trovate, NSE hits.

5.4 FASE 3 — ATTACCHI

La FASE 3 esegue attacchi mirati basati sui risultati della FASE 2. Ogni attacco è condizionato alla presenza del servizio corrispondente (porta aperta O finding nella FASE 2).

CVE exploitation via Metasploit

Se searchsploit ha trovato CVE nella FASE 1, exa-dune cerca i corrispondenti moduli Metasploit (fase A: search cve:XXXX-YYYY), poi esegue il comando check (non exploit!) su ciascuno. Se il check risponde "appears to be vulnerable", registra un finding CRITICAL.

Quick default IoT credentials

Prima del brute-force completo con hydra, exa-dune testa rapidamente 10 coppie di credenziali default comuni su ogni porta web aperta (HTTP basic auth con curl). Se una funziona, la salva nel credential store e procede.

Rate limiting detection

Prima del brute-force HTTP, exa-dune rileva rate limiting (HTTP 429/503) e backoff progressivo. Se rilevato, adatta i parametri hydra (-t 1 -W 5) per evitare blocchi.

Attacchi specializzati

  • IPMI cipher-zero: testa ipmitool con -C 0 (algoritmo di autenticazione nullo) — bypass auth completo su IPMI vulnerabili
  • SNMP write community: testa snmpset con community private/write/admin/rw/manager/secret — se accettato registra CRITICAL
  • TFTP file disclosure: tenta download di startup-config, running-config, config.bin, passwd, shadow, AudioCodes.ini, sip.conf e altri file sensibili
  • Web path traversal: 6 payload (../etc/passwd, URL encoding, double encoding, Windows win.ini) su ogni porta web
  • SMB vuln scan: NSE smb-vuln-ms17-010, smb-vuln-ms08-067, smb-vuln-cve-2020-0796
  • RTSP anonymous access: DESCRIBE senza credenziali su path comuni (/, /live, /stream, /h264/ch1/main/av_stream Hikvision, /cam/realmonitor Dahua, /axis-media/media.amp Axis, /h264Preview_01_main Reolink)
  • UPnP AddPortMapping injection: SOAP WANIPConnection per aprire port forwarding arbitrario
  • CoAP unauthenticated access: GET /.well-known/core + POST su ogni risorsa trovata
  • DNS zone transfer (AXFR): prova con hostname rilevato e domini locali (local, lan, home, internal)
  • Default creds vendor-specifiche: AudioCodes (Admin:Admin, admin:AudioCodes...), Commend, Axis (root:pass, root:axis...), Reolink (admin:), Hikvision (admin:12345, admin:Admin12345...), Dahua (admin:admin, 666666:666666...)

5.5 FASE 4 — REPORT PDF

La FASE 4 raccoglie tutti i risultati e genera il report finale:

  • Invoca exa-dune report OUTDIR_BASE --pdf
  • Copia il PDF in ~/Desktop/EXA-DUNE-Reports/REPORT_TARGET_TIMESTAMP.pdf
  • Invia notifica desktop tramite notify-send (se disponibile)
  • Crea archivio tar.gz compresso dell'intera directory output

Il report include: device profile, porte aperte, findings per severità (CRITICAL/HIGH/MEDIUM/LOW), credenziali trovate, CVE identificati, NSE hits, timeline operazioni.

5.6 Checkpoint e Resume

exa-dune salva automaticamente il progresso nel file .exa_checkpoint (una riga per ogni modulo/operazione completata). Se l'assessment viene interrotto, può essere ripreso con --resume.

# Avvio assessment normale
exa-dune auto 192.168.1.10 --output /root/pentest/assessment-ac/

# Se interrotto, riprendi da dove si era fermato
exa-dune auto 192.168.1.10 --resume /root/pentest/assessment-ac/

Il sistema legge .exa_checkpoint e salta i moduli già completati (skip visibile a schermo con il simbolo ↷). I moduli non completati vengono eseguiti normalmente. Se il checkpoint non esiste in --resume, il comando termina con errore.

exa-dune fw 192.168.1.20 --tech snmp-recon --snmp-community public

# Output: snmp_extra_ips.txt (IP interni), tcpConnTable (porte in ascolto)

Step 4: IPv6 bypass

# Molti firewall embedded non filtrano IPv6 link-local
# exa-dune calcola fe80:: e ripete lo scan
exa-dune fw 192.168.1.20 --tech ipv6

# Se trovato: exa-dune lancia service detection sulle porte bypassed
# e aggiorna open_tcp per la FASE 2
Nota sul bypass
Se tutte le 32 tecniche falliscono, exa-dune genera no_access_summary.txt con il registro di tutti gli approcci tentati e salta direttamente alla FASE 4 (report). Il report documenta l'impossibilità di accesso come finding informativo.
Scenario 4 — Discovery Rete Lab Completo

Procedura per scoprire tutti i device nel laboratorio e avviare assessment sequenziale.

Step 1: Network sweep

# Scopri tutti i device attivi nella rete VoIP
exa-dune network-sweep 192.168.1.0/24 --output /root/pentest/sweep-voip/

# Scopri tutti i device nella rete telecamere
exa-dune network-sweep 192.168.2.0/24 --output /root/pentest/sweep-cam/

Step 2: Assessment automatico su ogni device

# Leggi la lista host dal sweep e lancia auto per ognuno
for ip in $(grep "Host:" /root/pentest/sweep-voip/nmap_sweep.nmap | awk '{print $2}'); do
  exa-dune auto $ip --timeout 30 --output /root/pentest/lab-assessment/
done

Step 3: Analisi risultati aggregati

# Trova tutti i finding CRITICAL nel laboratorio
grep -r "\[CRITICAL\]" /root/pentest/lab-assessment/*/findings.txt

# Genera report per ogni device
for dir in /root/pentest/lab-assessment/exa-dune-*/; do
  exa-dune report "$dir" --pdf
done
